# E-inkダッシュボードサーバ 仕様書

## 1. プロジェクト概要

小型E-ink端末向けのダッシュボード画像を生成・配信するWebアプリケーション。端末はサーバの固定URLにアクセスして画像を取得し、画面に表示するだけのシンプルな構成とする。

### 1.1 想定される利用シーン

机の上やリビングに置かれたE-ink端末が、1時間に1回サーバに問い合わせて最新のダッシュボード画像を表示する。ユーザはスマホ等を使わずに、天気・予定・タスクをひと目で確認できる。

### 1.2 端末の制約

- **解像度**: 480×800 (縦画面)
- **色表現**: 4階調グレースケール (黒・濃灰・薄灰・白)
- **画像フォーマット**: BMP

```
│              │  ─────────────────────►    │                      │
│  E-ink端末   │                            │  Webアプリ(FastAPI)  │
│  (480×800)   │  ◄─────────────────────    │  Python              │
│ 4-level grayscale│ BMP画像 (480×800,4bit)  │                      │
└──────────────┘                            └──────────┬───────────┘
                                                       │
                              ┌────────────────────────┼────────────────────────┐
                              ▼                        ▼                        ▼
                    ┌──────────────────┐    ┌─────────────────┐    ┌─────────────────┐
                    │   気象庁JSON      │    │  calendar.json  │    │   todo.json     │
                    │  (天気予報)       │    │  (ローカル)      │    │   (ローカル)     │
                    └──────────────────┘    └─────────────────┘    └─────────────────┘
```

### 2.1 処理フロー

1. 端末がサーバにHTTP GETリクエスト
2. サーバが各データソースから情報を取得
   - 気象庁JSONから天気予報
   - `calendar.json`からカレンダー
   - `todo.json`からToDo
3. サーバがHTMLテンプレートにデータを差し込み
4. PlaywrightでHTMLをレンダリング → PNG画像化(中間)
5. Pillowで4階調パレット + Floyd-Steinbergディザリングを適用
6. 4bit BMPとしてエンコードしHTTPレスポンスとして返却

## 4. 画面仕様

### 4.1 画面サイズと色表現

- **解像度**: 480×800 (縦画面)
- **色**: 4階調グレースケール
  - 黒 `#000000` (palette index 0)
  - 濃灰 `#555555` (palette index 1)
  - 薄灰 `#AAAAAA` (palette index 2)
  - 白 `#FFFFFF` (palette index 3)
- **フォーマット**: 4bit BMP (16色パレット中4色のみ使用)
- **想定ファイルサイズ**: 約192KB

### 4.2 デザイン上の指針

4階調が使えるため、1bitモノクロより表現の幅が広い。CSS上は通常のグレースケールで自由にデザインしてよく、最終出力時に4階調へ自動量子化される。具体的な使い分けの目安:

- **黒**: 本文テキスト、強調、アイコンの主要部分
- **濃灰**: 補助テキスト、タイトル下線、表のヘッダ背景
- **薄灰**: 区切り線、無効状態、装飾要素、アイコンのシャドウ
- **白**: 背景

中間色を使ったグラデーションや影もFloyd-Steinbergディザリングで自然に表現される。

### 4.3 レイアウト

```
┌────────────────────────┐
│ ヘッダー                │  ~80px
│ (日付 / 曜日)           │
├────────────────────────┤
│                        │
│ 天気エリア              │  ~280px
│  ・今日の最高/最低気温   │
│  ・天気アイコン          │
│  ・時間帯ごとの予報      │
│                        │
├────────────────────────┤
│ カレンダー(今日) │ ToDo    │  ~440px
│  ・予定一覧      │  ・未完了│
│                 │   タスク │
└────────────────────────┘
```

### 4.4 各セクションの詳細

#### ヘッダー (~80px)
- 日付 (YYYY/MM/DD形式)
- 曜日 (日本語: 月/火/水/木/金/土/日)

#### 天気エリア (~280px)
- 取得地点: **東京地方** (予報区コード: 130010 / 気温地点コード: 44132)
- 今日の最高気温・最低気温
- 天気アイコン (晴れ/曇り/雨/雪などを気象庁天気コードからマッピング)
- 降水確率が提供される時間帯ごとの予報のうち、現在時刻を含む直近の時間帯から最大4件表示

#### カレンダー (下部左列)
- 今日の予定のみ表示
- ローカルJSONファイルから読み込み

#### ToDo (下部右列)
- 未完了タスクのみ表示 (`done: false`)
- 表示件数はスペースが許す限り (フォントサイズと領域から自動計算 or 固定上限を設定)

## 5. データ仕様

### 5.1 天気予報 (気象庁JSON)

エンドポイント: `https://www.jma.go.jp/bosai/forecast/data/forecast/130000.json`

利用する主なデータ:
- `timeSeries[0]`: 東京地方 (`130010`) の天気コード
- `timeSeries[1]`: 東京地方 (`130010`) の時間帯別降水確率
- `timeSeries[2]`: 東京 (`44132`) の気温
- 週間予報側の気温: 短期予報で最高/最低気温が空の場合の補完に利用

気象庁天気コード → アイコンマッピング (主要なもの):
- 100番台: 晴れ / 晴れ時々曇り
- 200番台: 曇り
- 300番台: 雨
- 400番台: 雪

### 5.2 カレンダー (calendar.json)

```json
[
  {
    "title": "歯医者",
    "start": "2026-04-27T10:00:00",
    "end": "2026-04-27T11:00:00",
    "location": "渋谷"
  },
  {
    "title": "チームMTG",
    "start": "2026-04-27T14:00:00",
    "end": "2026-04-27T15:00:00",
    "location": null
  }
]
```

フィールド:
- `title` (string, required): 予定タイトル
- `start` (ISO 8601, required): 開始日時
- `end` (ISO 8601, optional): 終了日時
- `location` (string, optional): 場所

サーバ側で「今日の予定」のみフィルタする。

### 5.3 ToDo (todo.json)

```json
[
  {
    "title": "請求書を送る",
    "done": false,
    "due": "2026-04-28"
  },
  {
    "title": "牛乳を買う",
    "done": false,
    "due": null
  },
  {
    "title": "ジムに行く",
    "done": true,
    "due": null
  }
]
```

フィールド:
- `title` (string, required): タスク名
- `done` (boolean, required): 完了フラグ
- `due` (date string, optional): 期限 (YYYY-MM-DD)

サーバ側で `done: false` のもののみ表示。期限が今日以前のものは強調表示するなど検討の余地あり。

## 6. APIエンドポイント

### 6.1 GET /dashboard.bmp

ダッシュボード画像を返却する唯一のエンドポイント。

**レスポンス**:
- 成功時: `200 OK` / `Content-Type: image/bmp` / 480×800 4bit BMP画像
- エラー時: `500 Internal Server Error`
  - 天気APIの取得失敗、ファイル読み込み失敗、レンダリング失敗など全てのエラーで500を返す
  - 端末側でエラーハンドリング(直前の画像表示維持など)を行う

**キャッシュ**: なし。リクエストごとに毎回生成する。

### 6.2 ヘルスチェック (任意)

`GET /health` で `200 OK` を返すだけのエンドポイントを用意してもよい。

## 7. 画像生成パイプライン

### 7.1 ステップ

1. **データ取得**
   - 気象庁JSON呼び出し (httpx)
   - `calendar.json` 読み込み
   - `todo.json` 読み込み
2. **HTMLレンダリング**
   - Jinja2テンプレートにデータ注入
   - 480×800のviewportでPlaywrightがChromiumを起動
   - スクリーンショットをPNGとして取得 (中間形式)
3. **4階調量子化 + ディザリング**
   - PillowでPNGを開く
   - グレースケール化 (`.convert("L")`)
   - 4階調パレットを定義し、Floyd-Steinbergディザリング付きで量子化
   - 結果は4色のみを含むパレット画像 (mode="P")
4. **BMPエンコード**
   - Pillowの `.save(buf, format="BMP")` で4bit BMPとして保存
   - Pillowはパレット色数が16以下なら自動で4bit BMPとして書き出す
5. **レスポンス返却**
   - `Response(content=bmp_bytes, media_type="image/bmp")`

### 7.2 量子化処理のサンプルコード

```python
from PIL import Image
import io

def to_4level_bmp(img: Image.Image) -> bytes:
    # 4階調パレット画像を作成
    palette_img = Image.new("P", (1, 1))
    palette = [
        0, 0, 0,         # index 0: 黒
        85, 85, 85,      # index 1: 濃灰
        170, 170, 170,   # index 2: 薄灰
        255, 255, 255,   # index 3: 白
    ] + [0] * (256 - 4) * 3
    palette_img.putpalette(palette)

    # グレースケール化 → 4階調量子化(Floyd-Steinbergディザリング)
    img_gray = img.convert("L")
    img_quantized = img_gray.quantize(
        palette=palette_img,
        dither=Image.Dither.FLOYDSTEINBERG,
    )

    # 4bit BMPとして書き出し
    buf = io.BytesIO()
    img_quantized.save(buf, format="BMP")
    return buf.getvalue()
```

### 7.3 Playwrightの起動コスト対策

Playwrightのブラウザ起動は数百ミリ秒〜1秒程度かかる。リクエストごとに起動すると遅延が大きいため、以下のいずれかを採用する:

- **アプリ起動時に1回だけブラウザを立ち上げ、コンテキストを使い回す** (推奨)
- リクエストごとに起動する (シンプルだが遅い)

FastAPIの `lifespan` ハンドラでブラウザを管理するのが綺麗。

## 10. 動作確認の流れ

1. ブラウザで `http://localhost:8080/dashboard.bmp` にアクセスして画像が返ることを確認
2. ブラウザでBMPがダウンロードされる場合は、画像ビューアで開いて見た目を確認
3. `data/todo.json` を編集して再アクセス、内容が反映されることを確認
4. 端末側のクライアント実装(画像取得 → 表示)と組み合わせて疎通確認
5. 4階調が正しく出ているか、ディザリングのザラつきが許容範囲か実機で確認
